# 双影密探 ShadowSleuth 技术架构

> 本文档基于项目需求推导，描述 **双影密探 ShadowSleuth** Android APK 的模块划分、数据流、数据模型以及性能与资源优化策略。语言尽量通俗，适合非技术背景的读者理解。

---

## 1. 架构目标

作为一款面向 Android 小白用户的工具，**双影密探 ShadowSleuth** 的技术架构需要满足：

- **搜索快**：扫描和匹配不能让用户等太久。
- **体积小**：APK 尽量精简，少依赖第三方库。
- **省资源**：不占用过多内存和电量，不后台常驻。
- **易维护**：模块职责清晰，便于后续迭代。

因此，整体采用轻量级原生架构，避免引入重量级框架。

---

## 2. 系统模块划分

整个 APP 可以分成 5 个主要模块：

| 模块 | 职责 | 通俗解释 |
|------|------|---------|
| **UI 层** | 展示界面、接收用户操作 | 用户看到的扫描页、结果页、搜索页、预览页 |
| **扫描器** | 遍历本地图片目录，读取图片元数据 | 像「图书馆管理员」一样，按目录把图片信息登记下来 |
| **匹配器** | 根据文件名或文件大小判断重复 | 像「查重机」一样，把相同名字或相同体积的图片归到一起 |
| **分组器** | 把匹配结果整理成可展示的分组 | 把重复图片两两配对，方便用户对比 |
| **权限处理器** | 申请和检查相册读取权限 | 敲门问用户能不能看相册，被拒绝时友好引导 |

### 2.1 模块关系图

```
用户点击「开始扫描」
    ↓
权限处理器检查相册权限
    ↓
扫描器遍历本地图片目录
    ↓
提取每张图片的元数据（路径、大小、时间、分辨率）
    ↓
匹配器按文件名 / 文件大小分组
    ↓
分组器生成两两对比的展示分组
    ↓
UI 层展示结果
    ↓
用户点击缩略图 → 预览原图和元信息
```

## 3. 数据流

一次完整扫描的数据流如下：

1. **触发扫描**：用户在扫描页点击「开始扫描」。
2. **权限检查**：如果没有读取权限，弹出引导，提示用户手动开启。
3. **目录遍历**：扫描器按预设目录（DCIM、Pictures、Download、微信/QQ 图片目录等）递归查找图片。
4. **文件过滤**：只保留 `jpg / png / webp / heic`，跳过视频、文档、安装包等。
5. **小图过滤**：忽略小于用户设定阈值的图片（默认 50KB）。
6. **元数据提取**：读取每张图片的：
   - 完整路径
   - 文件大小（字节）
   - 创建时间（毫秒时间戳）
   - 分辨率（宽 x 高）
   - 缩略图（按需生成，不预加载原图）
7. **匹配分组**：
   - 按文件名建立哈希表，相同名字归为一组。
   - 按文件大小建立哈希表，相同体积归为一组。
8. **合并结果**：去重后生成最终重复分组列表。
9. **展示结果**：UI 层以分组卡片形式展示，支持折叠/展开。

单图检索的数据流类似，只是扫描范围由「全部图片」变为「与样本图同名或同大小的图片」。

## 4. 数据模型

### 4.1 图片元数据（ImageMetadata）

```
ImageMetadata {
    id: 唯一标识（可由路径生成）
    path: 文件完整路径
    fileName: 文件名
    sizeBytes: 文件大小（字节）
    createdAt: 创建时间（时间戳）
    width: 图片宽度
    height: 图片高度
    mimeType: 图片类型，如 image/jpeg
}
```

### 4.2 重复分组（DuplicateGroup）

```
DuplicateGroup {
    groupId: 分组唯一 ID
    matchType: 匹配类型（文件名 / 文件大小）
    images: [ImageMetadata, ImageMetadata, ...]
}
```

### 4.3 扫描结果（ScanResult）

```
ScanResult {
    totalImages: 扫描的图片总数
    totalGroups: 重复分组数
    totalDuplicates: 重复图片总数
    groups: [DuplicateGroup, ...]
    scanDurationMs: 扫描耗时（毫秒）
}
```

## 5. 性能与资源优化策略

### 5.1 只读元数据，不加载原图

扫描阶段只读取图片的文件信息（路径、大小、时间、分辨率），不需要把整张原图加载到内存。只有用户点击缩略图进入预览时，才加载对应原图。

> 好处：大幅减少内存占用，扫描速度快，省电。

### 5.2 分块与异步扫描

- 扫描任务放在后台线程 / 协程中执行，不阻塞主界面。
- 图片目录可以分批处理，每处理完一批就更新一次进度条。
- 支持暂停 / 继续：用户切出 APP 或点击暂停时，保存当前进度，下次继续。

### 5.3 哈希表快速匹配

匹配阶段使用哈希表（HashMap / Dictionary）结构：
- 键：文件名或文件大小
- 值：相同键的图片列表

这样查找重复的时间复杂度接近 O(n)，而不是 O(n²)，即使图片很多也能快速完成。

### 5.4 缩略图按需生成与缓存

- 缩略图通过系统 MediaStore 或 ThumbnailUtils 获取。
- 生成后做内存缓存，避免同一图片重复解码。
- 退出 APP 或内存紧张时释放缓存。

### 5.5 小图过滤

扫描前按用户设置的阈值过滤掉极小的图片，减少后续匹配的数据量。

### 5.6 不后台常驻

扫描完成后立即释放资源，不保留后台服务或常驻进程，避免消耗电量和系统资源。

## 6. 技术选型原则

由于用户是 Android 开发新手，且产品要求轻量，建议遵循以下原则：

- **原生优先**：优先使用 Android 原生 API（MediaStore、ContentResolver、Coroutine）。
- **少依赖**：能不引入第三方库就不引入，控制包体积。
- **简单架构**：避免过度设计，不强制使用 MVVM/MVI 等复杂模式，但推荐将逻辑按模块拆分到独立类中。
- **逐步迭代**：先实现核心扫描和展示，再逐步优化体验。

---

## 7. 安全与隐私

- **无网络请求**：APP 不申请任何网络权限，不上传图片。
- **只读模式**：不申请删除/修改存储权限，仅在只读模式下展示结果。
- **数据不落盘**：扫描结果可以缓存在内存中，不强制保存到本地数据库，减少隐私风险。

## 8. 后续可扩展点（可选）

- 支持更多图片格式（如 BMP、RAW）。
